# Документация к Лабораторной работе №4: SQL-инъекции

Данный документ содержит полное описание разработанного приложения, принцип работы уязвимостей, методы защиты и список потенциальных вопросов для успешной сдачи лабораторной работы преподавателю.

---

## 1. Описание архитектуры приложения
Приложение представляет собой легковесный веб-сервер, написанный на платформе **Node.js** с использованием фреймворка **Express**. В качестве базы данных используется **SQLite**, которая разворачивается прямо в оперативной памяти (`:memory:`) при каждом запуске сервера. Это идеальный вариант для тестирования.

При запуске сервера в БД автоматически создается таблица `users` и заполняется тестовыми данными (пользователь `admin` и `user1`).

---

## 2. Разбор уязвимого кода (Часть 1)
Уязвимый эндпоинт доступен по адресу `/search`.

**Уязвимый участок кода:**
```javascript
const query = `SELECT * FROM users WHERE username = '${username}'`;
db.all(query, ...);
```

**В чем проблема?**
Здесь используется прямая конкатенация (склеивание) строк. Данные, которые вводит пользователь (`username`), напрямую вставляются в тело SQL-запроса без какой-либо проверки. База данных не может отличить, где заканчивается логика запроса, задуманная программистом, и где начинаются данные пользователя. Этим пользуется злоумышленник, передавая в качестве имени пользователя спецсимволы (кавычки), чтобы «сломать» синтаксис и дописать свой собственный SQL-код.

---

## 3. Разбор атак (Часть 2)

### Атака 1: Обход авторизации / Вывод всех данных
**Ввод злоумышленника:** `' OR 1=1 --`
**Как запрос выглядит для базы данных:**
```sql
SELECT * FROM users WHERE username = '' OR 1=1 --'
```
**Объяснение:**
1. Одинарная кавычка `'` закрывает строку, которую ожидал программист.
2. Оператор `OR 1=1` всегда возвращает `TRUE` (истину).
3. Символы `--` означают начало комментария в SQL. Все, что идет после них (оставшаяся кавычка от оригинального запроса), игнорируется базой данных.
**Итог:** Запрос выводит абсолютно все записи из таблицы `users`.

### Атака 2: UNION-атака (Извлечение скрытых данных)
**Ввод злоумышленника:** `' UNION SELECT id, username, password, is_admin FROM users --`
**Как запрос выглядит для базы данных:**
```sql
SELECT * FROM users WHERE username = '' UNION SELECT id, username, password, is_admin FROM users --'
```
**Объяснение:**
Оператор `UNION` объединяет результаты двух разных `SELECT` запросов в одну таблицу вывода. Злоумышленник берет пустой результат из первой части (`username = ''`) и "приклеивает" к нему всю таблицу целиком во второй части, получая доступ к паролям и статусу администратора.

## 4. Разбор защиты (Часть 3)
Защищенный эндпоинт доступен по адресу `/search-secure`. Для защиты применяется эшелонированный подход (два уровня):

### Уровень 1: Валидация входных данных (Белые списки)
```javascript
if (!/^[a-zA-Z0-9]+$/.test(username)) { ... }
```
Перед тем как вообще обращаться к базе данных, мы проверяем введенную строку с помощью регулярного выражения. Оно разрешает использовать **только** английские буквы (большие и маленькие) и цифры. Любые спецсимволы (кавычки, тире, пробелы) будут отброшены, и сервер вернет ошибку `400 Bad Request`.

### Уровень 2: Параметризованные запросы
```javascript
const query = "SELECT * FROM users WHERE username = ?";
db.all(query, [username], ...);
```
Вместо склеивания строк используется знак вопроса `?` (плейсхолдер). 
При таком подходе база данных сначала компилирует структуру SQL-запроса (строит план выполнения), и только потом вставляет данные на место знака `?`. Даже если переменная `username` содержит кавычки или команду `OR`, база данных воспримет их просто как текст (буквальное значение), а не как часть SQL-команды. Выполнить инъекцию становится технически невозможно.

## 5. Дополнительное задание: Логирование атак
В коде реализован `middleware` (промежуточный обработчик), который проверяет все входящие URL-адреса на наличие характерных для SQL-инъекций сигнатур (ключевых слов).
```javascript
const sqliPattern = /(\%27)|(\')|(\-\-)|(\%23)|(#)|(UNION)/i;
```
Если сервер замечает в URL кавычки, двойное тире или слово UNION, он пропускает запрос дальше (чтобы дать сработать защите из Части 3), но выводит в консоль тревожное сообщение (IP адрес атакующего, время и сам вредоносный URL). В реальном приложении эти логи отправляются администратору безопасности или в системы блокировки (Fail2Ban).

## 6. ВОПРОСЫ ДЛЯ ЗАЩИТЫ ЛАБОРАТОРНОЙ РАБОТЫ (FAQ)

**Вопрос 1: Что такое SQL-инъекция?**
*Ответ:* Это один из самых опасных видов уязвимостей веб-приложений. Она возникает, когда недобросовестный пользователь вводит в поля формы (или в URL) специальные символы и SQL-команды, которые меняют логику исходного запроса к базе данных. Это позволяет злоумышленнику читать, изменять или удалять закрытые данные.

**Вопрос 2: Зачем в конце SQL-инъекции ставится двойное тире (`--`)?**
*Ответ:* В языке SQL двойное тире означает начало однострочного комментария. Хакеры используют его, чтобы «отсечь» и закомментировать ту часть оригинального запроса, которая была написана программистом после вставляемой переменной. Без комментария запрос мог бы завершиться синтаксической ошибкой (например, из-за лишней оставшейся кавычки).

**Вопрос 3: Главное условие для проведения UNION-атаки?**
*Ответ:* Количество столбцов (и желательно их типы данных), возвращаемых в первом `SELECT` запросе, должно точно совпадать с количеством столбцов во втором `SELECT` запросе после слова `UNION`. В нашем коде `SELECT *` запрашивает 4 столбца, поэтому хакер тоже запрашивает 4 столбца: `id, username, password, is_admin`.

**Вопрос 4: Как именно защищают параметризованные запросы (prepared statements)? Почему они работают?**
*Ответ:* При использовании параметризованных запросов логика (SQL-шаблон) и данные пользователя отправляются драйверу базы данных *отдельно*. База данных заранее строит "дерево" выполнения запроса, и когда приходят данные, она подставляет их строго как значения, не пытаясь заново их парсить или интерпретировать как SQL-код.

**Вопрос 5: Зачем мы делали валидацию регулярными выражениями, если параметризованные запросы и так спасают от инъекций?**
*Ответ:* Это принцип эшелонированной защиты (Defense in Depth). 
Во-первых, если мы ждем логин пользователя, нет смысла вообще обращаться к базе данных, если туда ввели спецсимволы — мы отсекаем мусор еще на уровне логики сервера, экономя ресурсы.
Во-вторых, валидация защищает от других типов атак (например, XSS — межсайтовый скриптинг, если логин потом выводится на экран).

**Вопрос 6: Зачем логировать попытки атак (Доп. задание)?**
*Ответ:* Логирование позволяет вовремя обнаружить, что приложение подвергается сканированию уязвимостей. Зная IP-адрес злоумышленника, администратор может добавить его в черный список (Firewall) еще до того, как хакер найдет реальную уязвимость на сервере.
